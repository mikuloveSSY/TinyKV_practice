## 问题一

#### cf是在badger之上实现的。有没有考虑一定的单badger，为什么不可以一个cf一个单独的badger。是否有什么性能差异。要修改成一个cf一个badger的话，需要怎么修改代码？

> 这是架构选择问题——两种设计各有利弊，前者共享一份磁盘 I/O 和内存但需要前缀翻译，后者隔离更干净但管理多个数据库实例开销大。还要问如果改成多实例方案，对现有代码的改动范围有多大。本质是在考察 **对设计权衡的理解** 。

### 解答

#### 1. 为什么选择"一个 badger + 前缀"，而不是"一个 CF 一个 badger"？

**核心约束：跨 CF 原子写入。**

Project 4 的事务要求一次请求中同时写入三列族，这三条操作必须原子执行——要么全成功，要么全失败。如果三个 CF 是三个独立的 badger 实例，各自有独立的事务，无法保证跨实例的原子性。而一个 badger 实例内，一个 `db.Update` 可以同时写入多个 key，天然原子。

**次要约束：内存开销。**

每个 badger 实例启动后分配独立的内部结构：MemTable（写缓冲）、Block Cache（读缓存）、LSM 索引结构。方案 A（当前，一个 badger）只需一份；方案 B（三个 badger）需要三份，约三倍内存占用。

#### 2. 性能差异

|              | 一个 badger + 前缀 | 一个 CF 一个 badger |
| ------------ | :----------------: | :------------------: |
| 内存占用     |      一份缓存      |       三份缓存       |
| 磁盘 I/O     |      一条 WAL      | 三条 WAL（各自刷盘） |
| 写性能       |  顺序写，合并刷盘  |   三次写入各自刷盘   |
| 读性能       |        相当        |         相当         |
| 跨 CF 原子写 |      天然支持      |       无法实现       |

#### 3. 要改成"一个 CF 一个 badger"需要怎么改？

**存储层：** `StandAloneStorage` 从单个 `*badger.DB` 改成 `map[string]*badger.DB`，所有读写按 CF 分发到对应实例。`Write()` 跨 CF 时无法保证原子性。

**engine_util 层：** 前缀拼接/截断逻辑全删，`PutCF`、`GetCF` 只做转发。

**致命问题：** 即使改完，P4 跨 CF 原子事务仍然无法实现。如果用两阶段提交等外部协调器，复杂度远超前缀方案。这正是选择当前设计的根本原因——正确性优先。

## 问题二

#### 现在client和sever是在同一个进程不同线程，还是不同进程还是说不同协程？

> 这个问题的核心是想搞清楚 P1 的运行时拓扑的调用运行流——你写的 `raw_api.go` 被谁、以什么方式调用。本质是在考察 **对测试框架和 RPC 调用机制的理解** 。

### 解答

#### 前置概念：进程、线程、协程

**进程**：操作系管理系统资源（内存、文件句柄等）的独立单位。每个进程有自己独立的地址空间，进程之间不能直接共享内存——通信需要 IPC 机制（管道、共享内存、网络等）。一个 Go 程序就是一个进程。

> 为了保护：每个进程有独立的地址空间。假设两个程序VS Code 和 Chrome。如果它们共享内存，VS Code 写了个野指针可能导致Chrome 跟着崩了。进程的独立地址空间就是防火墙。

**线程**：进程内部的执行单元。同一进程内的多个线程**共享同一块内存地址空间**，可以直接读写相同变量，但也因此需要加锁保护。一个进程可以有多个线程，CPU 轮流调度它们执行。

> 为了利用多核：一个线程只跑一个 CPU 核。同进程内多线程不仅可以各自用一个核并行跑，而且共享内存，数据不用搬来搬去。

**协程（goroutine）**：Go 语言层面的轻量级执行单元，比线程轻得多（一个 goroutine 初始栈只需 ~2KB，线程要 ~1MB）。goroutine 由 Go 运行时调度到 OS 线程上执行，多个 goroutine 可以跑在同一个线程上。goroutine 之间通过 channel 通信。（*即线程要处理的任务*）

> 为了降成本：OS 线程切换要进内核，成本高。Go 运行时自己调度 goroutine，从而做到用少量线程处理成千上万的并发任务，降低线程切换的代价。

三者关系：

```text
进程
  ├── 线程 A ── goroutine 1, goroutine 2, goroutine 3 ...
  ├── 线程 B ── goroutine 4, goroutine 5 ...
  └── 内存空间（所有线程共享）
```

#### 回到 P1

问题问的三种场景翻译过来就是：

- "同一个进程不同线程"：client 和 server 跑在一个进程里，但 client 在一个线程上调 RPC，server 在另一个线程上处理请求
- "不同进程"：client 和 server 各自是独立的可执行程序，通过网络通信
- "不同协程"：client 在 goroutine A 里调函数，server 在 goroutine B 里处理，但共享同一进程

P1 的测试**既不是多线程、也不是多进程、也不是不同 goroutine**——就是最普通的直接函数调用。

从测试代码可以看到（`server_test.go:65`）：

```go
server := NewServer(s)               // 直接创建 Server 实例
resp, err := server.RawGet(nil, req) // 直接调你写的 handler
```

整个过程都在**同一个进程、同一个 goroutine** 中。没有网络监听、没有 gRPC 序列化、没有开任何协程。测试相当于把它当成一个普通的 Go 对象，直接调它的方法。

而 P2 完全不同——测试会启动 5 个节点组成的 Raft 集群（`NewTestCluster(5)`），每个节点拥有完整的 badger + Raft 模块 + gRPC 通信，等待选举完成后启动多个 goroutine 作为模拟客户端持续发送请求，还可注入随机丢包、节点崩溃重启、网络分区等故障，最终检查数据一致性。P1 的测试只是纯粹在进程内验证逻辑——你写函数，测试直接调，和单元测试没区别。

## 问题三

#### engine_util提供了两种函数，一个是Key()、Value()，一种是KeyCopy()、ValueCopy()，这两种有什么区别？

> 前者可能返回 badger 内部内存的引用（你只管读、别修改、别持有超过事务生命周期），后者是明确拷贝一份安全的副本给你。本质是在考察 **对 badger 内存模型和 GC/指针安全的理解** 。

### 解答

核心区别在于**内存所有权**：`Key()`/`Value()` 返回的是 badger 内部内存的引用（零拷贝），`KeyCopy()`/`ValueCopy()` 返回的是独立副本。

#### 源码关键部分

`Key()` 零拷贝返回内部字段：

```go
func (item *Item) Key() []byte { return item.key }
// 注释："Key is only valid as long as transaction is valid."
```

`Value()` 两种情况——小 value 零拷贝，大 value 从 vlog 读：

```go
func (item *Item) Value() ([]byte, error) {
    if item.meta & bitValuePointer > 0 {
        return item.db.blobManger.read(...)   // 大值：从磁盘拷贝
    }
    return item.vptr, nil                     // 小值：零拷贝，直接返回内部指针
}
// 注释："Value might change once discard or commit is called."
```

`KeyCopy()`/`ValueCopy()` 在直接版基础上加一层 `y.SafeCopy`，保证独立：

```go
func (item *Item) ValueCopy(dst []byte) ([]byte, error) {
    buf, _ := item.Value()       // 先取
    return y.SafeCopy(dst, buf)  // 再拷一层
}
```

#### 区别总结

| 对比项           |       Key() / Value()       | KeyCopy() / ValueCopy() |
| ---------------- | :--------------------------: | :---------------------: |
| 内存分配         |            零拷贝            |       分配新内存       |
| 指向 badger 内部 | 是（badger内存缓存的原数据） |     否（独立副本）     |
| 事务关闭后有效   |              否              |           是           |
| 可修改           |             不可             |          可以          |

> `Value()` 在小 value 时是零拷贝，大 value 时 badger 被迫从 vlog 拷贝，但**语义上仍不能假设它安全**——官方注释明确警告 discard/commit 后值可能改变。

#### 实际使用

P1 的 `RawScan` 里用直接版能通过测试，是因为测试框架收到响应后立即用 protobuf 序列化（即将 Go 结构体转成可在网络上传输的字节流——这个过程会逐字节读取 KvPair 中的 Key/Value 并拷进网络缓冲区，是一次真正意义上的数据拷贝），等 `defer reader.Close()` 触发 `txn.Discard()` 时，数据早已被 protobuf 消费完成。**严格来说直接返回 badger 内部引用是不安全的**——`[]byte` 赋值只拷贝切片头（指针+长度），底层数据还是同一块 badger 内存，测试能过只是序列化碰巧兜了底。如果出现循环中持有了 Key/Value、defer 已经 Close 了 reader、之后再访问数据的情况，就会踩到已释放的内存。

正确做法：需要跨事务持有数据时，用 `KeyCopy` / `ValueCopy` 显式拷贝。
